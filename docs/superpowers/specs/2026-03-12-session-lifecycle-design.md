# Session Lifecycle Design

**Date:** 2026-03-12
**Status:** Approved

---

## Background

When a message arrives from a third-party platform (Feishu, DingTalk, etc.), it carries a `sessionKey` that uniquely identifies the conversation context (e.g., `feishu:chatID:userID`). The system must ensure that:

- **Bee (brain)** always works within the same Claude session for a given `sessionKey`, maintaining conversation context across ticks.
- **Each worker** always works within the same Claude session for a given `sessionKey`, maintaining task execution context.
- A `/clear` command resets all session context for that `sessionKey`, causing both bee and workers to start fresh sessions.

Currently, session state is derived on-the-fly from the latest row in `platform_messages`, which has no per-agent tracking and no support for bee's session continuity. This design introduces a dedicated `session_contexts` table as the single source of truth for session state.

---

## Lifecycle Rules

### SessionKey Uniqueness

`sessionKey` is the immutable identifier for a conversation across all platform messages. All session decisions are scoped to `sessionKey`.

### Rule 1: Bee Brain

Bee works under the same `session_id` for a given `sessionKey` until a `/clear` command is received. Bee must **not** mix messages from different `sessionKey` values into a single invocation.

### Rule 2: Workers

Each worker works under the same `session_id` for a given `sessionKey` until a `/clear` command is received. After `/clear`, the next invocation generates a new `session_id`.

### Rule 3: Clear

A `/clear` command resets session context for **all agents** (bee + all workers) under that `sessionKey`.

## Database Schema

### New Table: `session_contexts`

```sql
CREATE TABLE session_contexts (
    session_key  TEXT     NOT NULL,
    agent_id     TEXT     NOT NULL,  -- "bee" or a worker_id (UUID)
    session_id   TEXT     NOT NULL,  -- Claude session ID for --session-id / --resume
    updated_at   DATETIME NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    PRIMARY KEY (session_key, agent_id)
);
```

**Design decisions:**
- Primary key `(session_key, agent_id)` enables atomic upsert with no prior read.
- `session_id` is a UUID generated by the application before invoking Claude CLI, passed via `--session-id` (first time) or `--resume` (subsequent times).
- No `execution_id` stored here ‚Äî bee has no execution concept; this table is agent-agnostic.
- `/clear` deletes all rows for `session_key`, resetting bee and all workers atomically.

### Store Layer Interface

```go
// UpsertSessionContext writes or updates the session_id for (sessionKey, agentID).
func (s *Store) UpsertSessionContext(ctx context.Context, sessionKey, agentID, sessionID string) error

// GetSessionContext returns the current session_id for (sessionKey, agentID).
// Returns ("", nil) if no session exists yet.
func (s *Store) GetSessionContext(ctx context.Context, sessionKey, agentID string) (string, error)

// ClearSessionContexts deletes all session_contexts rows for sessionKey.
func (s *Store) ClearSessionContexts(ctx context.Context, sessionKey string) error
```

## Component Changes

### 1. Feeder (Bee Brain)

**Problem:** `tick()` currently batches all `sessionKey` values into one bee invocation, violating session isolation.

**Change:** Group `regularMsgs` by `sessionKey` and process each group in a separate goroutine with its own bee invocation.

**Session logic per group:**
```
1. GetSessionContext(sessionKey, "bee")
   ‚îú‚îÄ‚îÄ exists  ‚Üí pass existing sessionID with --resume
   ‚îî‚îÄ‚îÄ missing ‚Üí generate uuid.New() as sessionID, pass with --session-id

2. runner.Run(ctx, workDir, prompt, sessionID, isResume)

3. success ‚Üí UpsertSessionContext(sessionKey, "bee", sessionID)
   failure ‚Üí rollback messages; do NOT update session_contexts
             (preserves last known good session_id for retry)
```

**`BeeRunner` interface update:**
```go
type BeeRunner interface {
    Run(ctx context.Context, workDir, prompt, sessionID string, resume bool) error
}
```

**Concurrency:** One goroutine per `sessionKey` in a batch. Goroutines for different `sessionKey` values run concurrently; messages within the same `sessionKey` are processed sequentially (one bee call per tick per `sessionKey`).

### 2. Dispatcher (Worker Session Lookup)

**Problem:** Currently uses `GetSession(sessionKey)` which derives session from `platform_messages`. This is replaced by `session_contexts`.

**Change:** Before dispatching a task to a worker, look up `session_contexts` instead of `platform_messages`.

**Session logic:**
```
1. GetSessionContext(sessionKey, workerID)
   ‚îú‚îÄ‚îÄ exists  ‚Üí CreateWithSessionID(workerID, input, existingSessionID) + --resume
   ‚îî‚îÄ‚îÄ missing ‚Üí Create(workerID, input) generates new UUID session_id  + --session-id

2. After execution completes successfully:
   ‚Üí UpsertSessionContext(sessionKey, workerID, execution.SessionID)
```

**`GetSession()` deprecation:** `MessageStore.GetSession()` can be removed once dispatcher no longer calls it.

### 3. Clear Command Handling

**Current:** Dispatcher evicts queues matching `sessionKey|*` and sends reply.

**Add:** Call `ClearSessionContexts(sessionKey)` before (or alongside) queue eviction.


## Migration

Add as a new migration in `store/db.go`:

```go
`CREATE TABLE IF NOT EXISTS session_contexts (
    session_key  TEXT     NOT NULL,
    agent_id     TEXT     NOT NULL,
    session_id   TEXT     NOT NULL,
    updated_at   DATETIME NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%fZ','now')),
    PRIMARY KEY (session_key, agent_id)
)`,
```

No data migration required ‚Äî existing sessions simply start fresh after deployment.

---

## Out of Scope

- Persistence of bee's `session_id` across server restarts is handled automatically by the `session_contexts` table surviving restarts.
- Cross-platform sessionKey collision is already prevented by the platform prefix in sessionKey format (e.g., `feishu:...`).
- Task routing (which worker handles which task) is unchanged ‚Äî the dispatcher continues to use `task.worker_id` to determine agent identity.

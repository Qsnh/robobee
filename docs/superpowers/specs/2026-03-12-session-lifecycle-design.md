# Session Lifecycle Design

**Date:** 2026-03-12
**Status:** Approved

---

## Background

When a message arrives from a third-party platform (Feishu, DingTalk, etc.), it carries a `sessionKey` that uniquely identifies the conversation context (e.g., `feishu:chatID:userID`). The system must ensure that:

- **Bee (brain)** always works within the same Claude session for a given `sessionKey`, maintaining conversation context across ticks.
- **Each worker** always works within the same Claude session for a given `sessionKey`, maintaining task execution context.
- A `/clear` command resets all session context for that `sessionKey`, causing both bee and workers to start fresh sessions.

Currently, session state is derived on-the-fly from the latest row in `platform_messages` via `GetSession()`, which has no per-agent tracking and no bee session continuity. Additionally, the feeder batches messages from all `sessionKey` values into a single bee invocation, violating session isolation. This design introduces a dedicated `session_contexts` table as the single source of truth for session state, refactors the feeder to process each `sessionKey` independently, and replaces the dispatcher's session-lookup mechanism.

---

## Lifecycle Rules

### Rule 1: Bee Brain

Bee works under the same `session_id` for a given `sessionKey` until a `/clear` command is received. Bee must **not** mix messages from different `sessionKey` values into a single invocation — each `sessionKey` gets its own bee invocation.

### Rule 2: Workers

Each worker works under the same `session_id` for a given `sessionKey` until a `/clear` command is received. After `/clear`, the next invocation generates a new `session_id`.

### Rule 3: Clear

A `/clear` command resets session context for **all agents** (bee + all workers) under that `sessionKey`.

## Database Schema

### New Table: `session_contexts`

```sql
CREATE TABLE session_contexts (
    session_key  TEXT    NOT NULL,
    agent_id     TEXT    NOT NULL,  -- "bee" (constant) or worker_id (UUID)
    session_id   TEXT    NOT NULL,  -- Claude session UUID for --session-id / --resume
    updated_at   INTEGER NOT NULL,  -- Unix milliseconds (consistent with rest of schema)
    PRIMARY KEY (session_key, agent_id)
);
```

**Design decisions:**
- Primary key `(session_key, agent_id)` enables atomic upsert. The PK index prefix also covers the `ClearSessionContexts` scan by `session_key` alone — no additional index needed.
- `session_id` is the Claude session UUID, pre-generated by the application before invoking Claude CLI. It is the same value stored in `worker_executions.session_id`. It is passed via `--session-id` (first call) or `--resume` (subsequent calls).
- `updated_at` is `INTEGER NOT NULL` (Unix milliseconds via `time.Now().UnixMilli()`), consistent with all other timestamp columns in the schema.
- The constant string `"bee"` is used as `agent_id` for all bee invocations. This must be a named constant (`const BeeAgentID = "bee"`) to prevent divergence.
- `/clear` deletes all rows for `session_key` in one statement, atomically resetting bee and all workers.
- No data migration required — existing sessions start fresh after deployment.
- When a worker is deleted, its `session_contexts` rows (keyed by `worker_id` as `agent_id`) become orphaned but harmless. Cleanup on worker deletion is not required for correctness.

### Store Layer Interface

```go
const BeeAgentID = "bee"

// UpsertSessionContext writes or updates the session_id for (sessionKey, agentID).
func (s *Store) UpsertSessionContext(ctx context.Context, sessionKey, agentID, sessionID string) error

// GetSessionContext returns the current session_id for (sessionKey, agentID).
// Returns ("", nil) when no row exists (first-ever call for this pair) — normal case, not an error.
// Returns non-nil error only on database failure.
func (s *Store) GetSessionContext(ctx context.Context, sessionKey, agentID string) (string, error)

// ClearSessionContexts deletes all session_contexts rows for sessionKey.
func (s *Store) ClearSessionContexts(ctx context.Context, sessionKey string) error
```

## Component Changes

### 1. Feeder (Bee Brain)

**Scope of change:** The entire body of `tick()` is structurally refactored. The current implementation processes all regular messages as a single batch with a single bee invocation. After this change, `tick()` groups messages by `sessionKey` and runs one bee invocation per group in a separate goroutine. This is a rewrite of the fan-out logic, not an additive change.

**WorkDir and CLAUDE.md:** `WriteCLAUDEMD` is called once before the fan-out (not per-goroutine). All bee invocations share the same `workDir` and persona file — no race condition.

**BeeRunner interface change (breaking):**

Current signature:
```go
type BeeRunner interface {
    Run(ctx context.Context, workDir, prompt string) error
}
```

New signature:
```go
type BeeRunner interface {
    Run(ctx context.Context, workDir, prompt, sessionID string, resume bool) error
}
```

**BeeProcess.Run change:** `BeeProcess` must be updated to pass the appropriate CLI flags, mirroring the pattern in `ClaudeRuntime.Execute` (`worker/claude_runtime.go` lines 30-36):

```go
func (p *BeeProcess) Run(ctx context.Context, workDir, prompt, sessionID string, resume bool) error {
    args := []string{
        "--dangerously-skip-permissions",
        "--output-format", "stream-json",
        "--mcp-server", "robobee=" + p.mcpURL,
        "--mcp-api-key", p.apiKey,
        "-p", prompt,
    }
    if sessionID != "" {
        if resume {
            args = append(args, "--resume", sessionID)
        } else {
            args = append(args, "--session-id", sessionID)
        }
    }
    // ... rest unchanged
}
```

**Session logic per sessionKey group:**

```
sessionID, err = GetSessionContext(sessionKey, BeeAgentID)

if sessionID == "" {        // no prior session for this sessionKey
    sessionID = uuid.New().String()
    resume = false          // use --session-id (establish new session)
} else {
    resume = true           // use --resume (continue existing session)
}

runner.Run(ctx, workDir, prompt, sessionID, resume)

success → UpsertSessionContext(sessionKey, BeeAgentID, sessionID)
failure → rollback this sessionKey's messages only
          do NOT update session_contexts (preserves last known good session_id)
```

**Rollback scope:** Each sessionKey group is rolled back independently. A failure in one group does not affect other groups in the same tick.

**Concurrency:** One goroutine per `sessionKey` per tick. `tick()` waits for all goroutines before returning (`sync.WaitGroup`).

**Recovery on restart:** `RecoverFeeding` resets `feeding` messages to `received` at startup. On the next tick, normal flow applies: if a `session_contexts` row exists, resume that session; if not, start fresh. No special recovery logic is needed for `session_contexts`.

### 2. Dispatcher (Worker Session Lookup)

**Scope of change:** The existing `GetSession(sessionKey)` + `ReplyExecution(executionID)` resume path is replaced entirely. `ReplyExecution` is no longer called by the dispatcher for the session-resume case — the dispatcher will call `CreateWithSessionID` directly (or a new `Manager` method that wraps it), using the `session_id` UUID retrieved from `session_contexts`.

**Interface changes on Dispatcher:**

`dispatcher.MessageStore` interface:
- Remove `GetSession(sessionKey string) (*platform.Session, error)`
- Add `GetSessionContext(ctx context.Context, sessionKey, agentID string) (string, error)`

`dispatcher.ExecutionManager` interface:
- Remove `ReplyExecution(ctx context.Context, executionID, message string) (model.WorkerExecution, error)`
- Add `ExecuteWorkerWithSession(ctx context.Context, workerID, input, sessionID string) (model.WorkerExecution, error)`
- Retain `ExecuteWorker` unchanged

The concrete `Manager.ReplyExecution` method is **retained** (not deleted) because the HTTP API handler may call it directly. It is simply removed from the `dispatcher.ExecutionManager` interface since the Dispatcher no longer uses it.

**Session logic:**

```
sessionID, err = store.GetSessionContext(sessionKey, workerID)

if sessionID == "" {
    // Fresh start: Create() generates a new UUID session_id internally
    exec = manager.ExecuteWorker(ctx, workerID, input)
    // ExecuteWorker → executionStore.Create → new UUID → --session-id
} else {
    // Resume: pass the stored session_id UUID directly
    exec = manager.ExecuteWorkerWithSession(ctx, workerID, input, sessionID)
    // new method: executionStore.CreateWithSessionID + launchRuntime(..., resume=true)
}
```

**New `Manager` method (or renamed path):** A new `ExecuteWorkerWithSession(ctx, workerID, input, sessionID string)` method is needed, analogous to the existing `ReplyExecution` but taking a `sessionID` directly instead of an `executionID`. This avoids the extra `executionStore.GetByID` lookup that `ReplyExecution` currently does.

**After execution completes (in `monitorExecution`):**
```
OutputDone (success) → UpsertSessionContext(sessionKey, workerID, execution.SessionID)
OutputError (failure) → do NOT update session_contexts
```

To call `UpsertSessionContext` in `monitorExecution`, the `sessionKey` must be passed into the monitoring goroutine at dispatch time (e.g., as a field on the execution record or as a closure parameter).

**`GetSession()` removal:** `MessageStore.GetSession()` is no longer called anywhere after this change and can be deleted.

**`ReplyExecution` retention:** `ReplyExecution` may be retained for the HTTP API (if any external caller uses it) or removed if it becomes dead code. This is a separate decision.

### 3. Clear Command Handling

**Addition to dispatcher clear path:**

```go
case "clear":
    if err := store.ClearSessionContexts(ctx, sessionKey); err != nil {
        log.Printf("dispatcher: clear session contexts: %v", err)
    }
    evictQueues(sessionKey)
    sendReply("✅ 上下文已重置")
```

**Clear race:** A worker execution may complete after `ClearSessionContexts` runs, causing `UpsertSessionContext` to re-create a stale row. The accepted approach is "last write wins" — the stale row will be overwritten by the next successful execution or cleared by the next `/clear`. Given that `/clear` is an explicit user action and the stale window is short, this is acceptable for now.

### 4. Test Updates

**`feeder_test.go` — breaking interface change:** `mockBeeRunner.Run` must be updated to match the new signature:

```go
func (m *mockBeeRunner) Run(_ context.Context, _, prompt, sessionID string, resume bool) error
```

**New feeder test cases required:**
- First tick for a sessionKey (no prior context): runner receives `resume=false`, a new non-empty `sessionID`; `UpsertSessionContext` is called after success.
- Subsequent tick for the same sessionKey: runner receives `resume=true` with the previously stored `sessionID`.
- Failed bee run: `UpsertSessionContext` is NOT called; messages are rolled back.
- Two different sessionKeys in one batch: each group runs independently; a failure in one does not affect the other.

**`dispatcher_test.go` (if exists):**
- Session resume path: `GetSessionContext` returns a non-empty `sessionID` → `ExecuteWorkerWithSession` is called, not `ExecuteWorker`.
- Fresh path: `GetSessionContext` returns `""` → `ExecuteWorker` is called.

## Data Flow Summary

```
Inbound message (sessionKey=K, content=X)
    │
    ▼
platform_messages (status=received)
    │
    ▼
Feeder.tick()  [REWRITTEN]
    WriteCLAUDEMD(workDir, persona)        ← once, before fan-out
    separate clearMsgs / regularMsgs
    group regularMsgs by sessionKey
    for each sessionKey K (concurrent goroutines, sync.WaitGroup):
        GetSessionContext(K, "bee") → sessionID
        sessionID=="" → new UUID, resume=false
        sessionID!="" → resume=true
        BeeProcess.Run(workDir, prompt, sessionID, resume)
        success → UpsertSessionContext(K, "bee", sessionID)
        failure → rollback K's messages only
    │
    ▼
tasks table (worker_id=W, linked to session_key=K via message_id)
    │
    ▼
Dispatcher  [UPDATED]
    GetSessionContext(K, W) → sessionID
    sessionID=="" → manager.ExecuteWorker(K, W, input)
    sessionID!="" → manager.ExecuteWorkerWithSession(K, W, input, sessionID)
    monitorExecution:
        OutputDone → UpsertSessionContext(K, W, execution.SessionID)
        OutputError → skip upsert

/clear (sessionKey=K)
    ClearSessionContexts(K)    ← deletes all rows for K (bee + all workers)
    evictQueues(K)
    sendReply("✅ 上下文已重置")
```

## Out of Scope

- Per-sessionKey isolated workDirs for bee (shared workDir is sufficient).
- Strict clear-race prevention via generation tokens (last-write-wins is acceptable).
- Cleanup of orphaned `session_contexts` rows on worker deletion.
- Changes to the HTTP API surface for `ReplyExecution`.
